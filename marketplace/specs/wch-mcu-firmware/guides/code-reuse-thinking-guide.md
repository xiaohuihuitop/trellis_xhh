# 代码复用思考指南

> 在 MCU 固件项目里，先判断"应不应该复用"，再决定"怎么写"。

---

## 写新代码前先检查

- 先搜有没有现成模块已经负责这类行为（`grep -r "关键词" .`）
- 先看是不是已有事件分支或 Task 模块可扩展
- 改常量、枚举、协议字段前，**先全局搜索所有引用**

---

## 本项目的复用边界

| 已有模块 | 负责的行为 | 扩展方式 |
|----------|-----------|----------|
| `xhh_Task_Flash` | 所有 Flash 读写 | 加字段进结构体，不要散写 Flash |
| `xhh_Event` | 多模块联动 | 加事件枚举 + case，不要跨 Task 直调 |
| `xhh_Mode` | 系统状态 | 加状态枚举 + Change，不要另建状态机 |
| `xhh_Task_<X>` 四件套 | 单一功能域 | 加 `_Set_*` / `_Apply_*`，不要在 main_task 堆逻辑 |
| `Flash_Erase/Read/Write` | Flash 原语 | 直接调，不要另封装一层 |

---

## 常见重复陷阱

| 陷阱 | 错误做法 | 正确做法 |
|------|----------|----------|
| 协议入口写业务状态更新 | 在 `BLE_HANDLE.c` 里改 motor/ui/timeout | 走事件层统一协调 |
| 多个 Task 各自写 Flash | Motor/BAT 各调 `Flash_Write` | 集中到 `xhh_Task_Flash` |
| 重复的命令拼包逻辑 | 多处复制 | 抽到协议层 |
| 重复的范围校验 | 多处写同一 min/max 检查 | 抽 `IS_Valid` 谓词 |
| 新建功能不建 Task 模块 | 在 `main_task.c` 堆逻辑 | 建 `xhh_Task_<X>` 四件套 |

---

## 推荐判断

- 已有边界清晰的模块存在 → 优先扩它
- 同一逻辑出现 3 次以上 → 考虑抽公共 helper
- 只有一次使用且改成抽象更难懂 → 不要过度抽象
- 改常量/枚举前 → **必须** grep 全局引用，避免"忘了改 X"bug

---

## 改常量前必做

```bash
# 搜所有引用，确认没有遗漏
grep -r "要改的值" .
grep -r "要改的宏名" .
```

这一个习惯能挡住大多数"忘了更新 X"的 bug。

---

## Checklist

- [ ] 搜过现成模块/函数
- [ ] 没有 copy-paste 该抽公共的逻辑
- [ ] 常量定义在一处
- [ ] 改常量/枚举前 grep 过全局引用
- [ ] 新功能考虑过扩展现有 Task 模块而非新建
