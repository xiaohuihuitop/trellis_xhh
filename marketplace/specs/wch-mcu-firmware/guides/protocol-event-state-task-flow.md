# 协议 → 事件 → 状态机 → Task 链路指南

> 这是本项目最核心的思考指南。拿到任何新需求，先沿这条链路推导"该落哪一层"，再动手。

---

## 链路全貌

```text
外部输入                 统一桥梁           系统节奏             执行单元
─────────────────────────────────────────────────────────────────────────
BLE 收数      ─┐
按键检测       ─┼─> xhh_Event_Trigger ─> xhh_Event_Handle ─> xhh_SYS_Handle ─> xhh_Task_*_Loop
硬件状态检测   ─┘    (写全局 Event_n)     (switch 分发)        (switch 状态机)    (各模块执行)
                                          ↓                      ↓
                                   xhh_SYS_Change        xhh_Task_*_Cmd/Set
```

每一层只做一件事：

| 层 | 职责 | 不做什么 |
|----|------|----------|
| 协议入口 | 帧解析、命令识别、参数拆解 | 不写业务流程、不直接改状态机 |
| 事件层 | 多模块联动协调 | 不做具体硬件操作 |
| 状态机层 | 系统节奏、状态推进 | 不做协议解析、不写 Flash |
| Task 层 | 单一功能域执行 | 不跨模块直接调别的 Task |

## 拿到新需求怎么推导

### 场景 A：BLE/App 下发一个新命令

1. **协议层**：在 `protocols.h` 加功能码宏，在 `BLE_HANDLE.c` 解析后调 `xhh_Event_Trigger`
2. **事件层**：在 `xhh_Event.h` 加事件枚举，在 `xhh_Event_Handle` 加 case
3. **事件 case 内**：判断需要联动哪些模块，调各 `xhh_Task_*_Set_*`，再 `xhh_SYS_Change` 切状态
4. **Task 层**：若新功能需要新硬件域 → 建 `xhh_Task_<X>` 模块；若是已有模块扩展 → 加 `_Set_*` / `_Apply_*`

**反模式**：在 `BLE_HANDLE.c` 里直接调 `xhh_Task_Motor_Set_*` + 改 UI + 改 timeout。这绕过了事件层，多模块联动会分叉。

### 场景 B：新增一个硬件功能（如加一个传感器）

1. **Task 层**：建 `xhh_Task_<Sensor>.c/.h`，四件套 + BSP static 封装
2. **聚合层**：注册到 `xhh_Task_ALL.h` / `xhh_Task.c`
3. **状态机层**：在合适状态的 Entry 里 `_Cmd(1)` 开启，PowerOFF 里 `_Cmd(0)` 关闭
4. **事件层**：若该传感器需要触发系统行为（如超阈值报警），在 Task Loop 内检测后 `xhh_Event_Trigger`
5. **持久化**：若需记忆配置，加进 `xhh_Task_Flash_user_data_t` 结构体

### 场景 C：改一个已有行为（如调整某状态的持续时间）

1. **状态机层**：定位 `xhh_SYS_Handle` 内对应状态的 Ing 阶段 `Loop_Count` 阈值
2. 只改阈值，不改链路结构
3. 若该时间需持久化 → 加进 Flash 结构体

### 场景 D：纯算法/计算（如时间换算、CRC）

1. 不属于任何业务层，放 `APP/` 下独立辅助文件（如 `utc_time/`）
2. 不走事件/状态机，被需要的模块直接调

## 判断"这逻辑该放哪"的决策树

```
这段逻辑...
│
├─ 解析外部协议帧？ ────────────────> 协议入口层
│
├─ 同时影响 ≥2 个模块？ ────────────> 事件层（加事件 + case）
│
├─ 决定系统该进什么状态？ ──────────> 状态机层（或事件 case 内调 Change）
│
├─ 操作单一硬件/单一业务域？ ──────> Task 层（对应模块）
│
├─ 中断里必须立刻做的事？ ──────────> 中断（只清标志/计数/极轻量输出）
│   └─ 见 isr-vs-main-loop.md
│
└─ 纯计算/无副作用？ ───────────────> APP/ 辅助文件
```

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 协议层直接改状态机 | 联动模块没同步更新 | 走事件层 |
| 事件 case 内只改了一半模块 | 状态分叉 | case 内全改或全不改 |
| 新硬件功能不建 Task 模块 | main_task 堆满逻辑 | 建四件套模块 + 注册 ALL |
| Task 模块间直接互调 | 耦合、难调度 | 通过事件或状态机协调 |
| 中断里做协议解析 | 时序崩、丢数据 | 中断只置标志，主循环处理 |

## 链路追踪练习

拿到一个 bug，沿链路反向追：

```
现象（如：按键后电机没响应）
  ↑
Task 层：xhh_Task_Motor_Loop 使能位开了吗？_Set_* 被调了吗？
  ↑
状态机层：xhh_SYS_n 在预期状态吗？xhh_SYS_Change 被调了吗？
  ↑
事件层：xhh_Event_Handle 里对应 case 走了吗？前置守卫挡了吗？
  ↑
协议/输入层：xhh_Event_Trigger 被调了吗？Event_n 值对吗？
```

用 `XHH_DEBUG` 打印 `e_h`（事件号）和 `s_h`（状态切换）能快速定位卡在哪一层。
